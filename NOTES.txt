The "Honey Pot" Real-Time Architecture Plan


  Phase 1: Environment & Dependency Setup
   * Step 1.1: Dependency Injection. Update pubspec.yaml to include supabase_flutter for the backend and http for the
     Quotes REST API.
   * Step 1.2: Initialization. Configure main.dart to initialize Supabase with your Project URL and Anon Key.
   * Step 1.3: Secure Storage. Integrate flutter_secure_storage to handle session persistence so users stay logged in.


  Phase 2: Cloud Schema & Storage (Supabase Side)
   * Step 2.1: SQL Tables. I will provide you with a SQL script to run in your Supabase SQL Editor. This will create:
       * profiles (linked to Auth).
       * groups (with join_code and total_pot).
       * group_members (with rotation_order).
       * contributions (status: pending/verified).
   * Step 2.2: Storage Buckets. Create payment-proofs and user-ids buckets in Supabase to host the images globally.
   * Step 2.3: Row Level Security (RLS). Set up policies so only group members can see the detailed transactions of
     their specific "Honey Pot."


  Phase 3: The Hybrid Data Layer (Service Refactoring)
   * Step 3.1: SupabaseService. Build a new service to handle all Cloud CRUD operations.
   * Step 3.2: Sync Logic (The Bridge). Modify DbService to act as a "Local Mirror."
       * Logic: When the app is online, fetch from Supabase and overwrite the SQLite cache. When offline, read only from
         SQLite.
   * Step 3.3: Real-time "Stream" Integration. Use supabase.from('groups').stream(...) in your ViewModels so the "Total
     Pot" on the UI jumps the moment a payment is verified.


  Phase 4: Authentication & Email Verification
   * Step 4.1: Supabase Auth. Replace local login/signup with supabase.auth.signUp.
   * Step 4.2: Verification Flow. Implement a "Waiting for Verification" UI state. The user is redirected to a screen
     telling them to check their email before they can access the Home Page.


  Phase 5: Core "Honey Pot" Logic Updates
   * Step 5.1: Randomized Rotation. Implement the "Start Group" logic on the server-side (or via the Creator's app) that
     shuffles the list of members and assigns their payout rounds fairly.
   * Step 5.2: The 20% Fee Logic. Update the payout calculation to deduct the 20% fee into a "Held" status, with a final
     "Cycle Completion" transaction that refunds it to all verified participants.
   * Step 5.3: REST API Feature. Implement the QuotesService using the http package to pull daily financial quotes to
     the Home View.
